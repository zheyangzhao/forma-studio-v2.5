# Forma Studio v3.0 Backlog 評估報告

評估日期：2026-04-29  
評估範圍：design-systems markdown、Comment mode、多格式匯出  
結論立場：v3.0 應做「低破壞、高交付價值」的增量升級；不建議改成 nexu-io/open-design 或 open-codesign 的整套架構。

資料來源：

| 類型 | 來源 |
|---|---|
| 本地規格 | `docs/SDD-v2.5-integration-upgrade.md`、`docs/PLAN-sprint-1.5.md` |
| 本地實作 | `desktop/app/utils/design_memory.py`、`web/forma-studio.html`、`desktop/app/widgets/image_edit_panel.py` |
| 本地狀態 | `README.md`、`DEVLOG.md`、`HANDOFF.md` |
| 外部參考 | `https://github.com/nexu-io/open-design`、`https://github.com/nexu-io/open-design/tree/main/design-systems`、`https://opencoworkai.github.io/open-codesign/`、`https://support.claude.com/en/articles/14604416-get-started-with-claude-design` |

## 一、摘要（給用戶 3 分鐘看完的執行版）

### 1.1 三項評估結論一表

| Backlog | 推薦 | ROI | 估計工時 | 主要收益 | 主要風險 | v3.0 立場 |
|---|---:|---:|---:|---|---|---|
| A. design-systems markdown | 有條件做 | 高 | 18-28h | 品牌記憶更完整，輸出更穩 | schema 膨脹、與既有 parser 不相容 | 做 b：擴充 v2.5 子集 |
| B. Comment mode | 部分做 | 中 | 36-58h | 降低局部修改成本 | DOM/圖片區域定位、版本同步、模型越界修改 | 先做 Web/HTML preview 版，不做桌面自動 mask |
| C. 多格式匯出 | 做 | 高 | 28-46h | 律師/教師/行銷有明確交付格式 | PDF/PPTX fidelity、跨平台工具鏈 | 先 PDF + Markdown，PPTX 第二階段 |

判斷核心：

- A 的價值在「把品牌規則變成可機讀 prompt context」，不是複製 71 個品牌資料庫。
- B 的價值在「精準回饋」，但 v2.5 目前主力仍是 prompt 生成與圖片 edit，不是完整 DOM artifact editor。
- C 的價值最直接，因為跨行業客戶本來就用 PDF、PPTX、Markdown 交付或交接。

### 1.2 v3.0 整體建議

v3.0 建議路線：

```text
v3.0
├── 必做
│   ├── DESIGN.md v2.5 schema 擴充：加入 tokens/components/motion/voice 子集
│   ├── Markdown export：prompt + metadata + source attribution
│   └── PDF export：優先服務提案、教材、客戶交付
├── 條件做
│   ├── Web Comment mode：只針對 HTML/preview artifact，feature flag 開啟
│   └── PPTX export：等 PDF/Markdown 穩定後做
└── 不做
    ├── 完整替換為 nexu-io/open-design 9-section schema
    ├── 桌面版自動 DOM hit-test comment mode
    └── ZIP/HTML export 作為 v3.0 必做項
```

### 1.3 估計工時與里程碑

| 里程碑 | 範圍 | 工時 | 驗收標準 |
|---|---|---:|---|
| M1 | DESIGN.md schema 擴充與 backward-compatible parser | 18-28h | 舊 6 區塊可讀；新子集可存取；既有 40 pytest 不退步 |
| M2 | Markdown + PDF export MVP | 28-46h | 可匯出 prompt、metadata、圖片；PDF 字型與分頁可控 |
| M3 | Web Comment mode prototype | 36-58h | iframe/preview 可 pin；comment 可轉成局部 patch prompt；可關 feature flag |
| M4 | PPTX export spike | 20-34h | 3-5 種 layout 可編輯；不承諾 1:1 視覺還原 |

**決策摘要：v3.0 不應追求功能大而全，應先做 A(b) + C(PDF/Markdown)，B 只做可控原型。**

## 二、A：design-systems markdown 借鑑評估

### 2.1 nexu-io 格式現況（schema 範例 + 71 systems 規模）

nexu-io/open-design 的公開 README 描述為 Apache-2.0、local-first、19 skills、71 brand-grade design systems。GitHub 頁面在 2026-04-29 讀取時顯示約 2k 級 star，資料夾頁顯示 298 forks / 2.5k stars，repo 首頁快取片段顯示 223 forks / 1.9k stars。由於 star 變動快，本報告採「約 2k 級關注度」而非鎖死單一數字。用戶提供的 1338 stars 可視為較早快照。

design-systems 目錄現況：

| 指標 | 數字/描述 |
|---|---|
| design systems | 71 |
| 手寫 starter | 2：`default`、`warm-editorial` |
| 匯入來源 | 69 product systems from `VoltAgent/awesome-design-md` / `getdesign` npm package |
| metadata | H1 title + `> Category: <name>` |
| README 宣稱 schema | 9 section：color、typography、spacing、layout、components、motion、voice、brand、anti-patterns |
| 實際用途 | dropdown 選品牌，active `DESIGN.md` 進入 system prompt |

schema 級別範例：

```markdown
# Design System Inspired by Cohere

> Category: AI & LLM
> Enterprise AI platform. Vibrant gradients, data-rich dashboard aesthetic.

## 1. Visual Theme & Atmosphere
...

## 2. Color System
...

## 3. Typography
...

## 4. Spacing & Layout
...

## 5. Components
...

## 6. Motion
...

## 7. Voice & Tone
...

## 8. Brand Signals
...

## 9. Anti-patterns
...
```

關鍵觀察：nexu-io 的 DESIGN.md 是「artifact generation context」，不是嚴格 design token compiler。它對 LLM 很友善，對 deterministic parser 需要額外約束。

### 2.2 v2.5 DESIGN.md 現況（schema 範例 + 6 區塊）

v2.5 的 `desktop/app/utils/design_memory.py` 已有可解析、可驗證、可序列化、可注入 prompt 的 6 區塊 schema。

```markdown
# Forma Studio · DESIGN.md

## Brand Identity
- project_name: 範例專案
- brand_name: ACME Corp
- industry: legal
- audience: 律師, 企業客戶
- tone_of_voice: 嚴肅高端

## Color Tokens

| token | value |
|---|---|
| primary | oklch(55% 0.18 250) |
| secondary | oklch(75% 0.10 90) |

## Typography
- heading: Noto Serif TC
- body: Noto Sans TC

## Visual Rules
- 不用 emoji 圖標
- 不用紫漸層

## Prompt Defaults
- size: portrait
- quality: high

## Negative Constraints
- watermark
- fake brand logo
```

現況能力：

| 能力 | v2.5 現況 |
|---|---|
| parser | `##` section split + bullet kv + markdown table |
| validation | brand_name 缺失警告、color token 空值警告 |
| serialization | `memory_to_markdown()` 可 round trip |
| prompt injection | `build_system_prompt()` / `apply_design_memory_to_prompt()` |
| GUI | 桌面版品牌記憶 tab 可編輯 |
| 測試基線 | DEVLOG 最新紀錄：40 pytest PASS，其中 design_memory 9 tests |

優點：小、穩、符合跨行業 prompt 工具定位。缺點：component、spacing、motion、voice 還沒有結構化欄位，無法描述完整設計系統。

### 2.3 差異對照表（schema / tooling / use case）

| 維度 | nexu-io/open-design | Forma Studio v2.5 | 判斷 |
|---|---|---|---|
| schema 大小 | 約 9 section | 6 section | v2.5 較可控 |
| design system 數量 | 71 | 12 品牌參考 + DESIGN.md 自填 | nexu 規模大，但授權/品牌真實性需控管 |
| metadata | H1 + Category blockquote | Brand Identity kv | 兩者可映射 |
| tokens | color/type/spacing/components/motion 等 | color + typography + prompt defaults | v2.5 缺 components/spacing |
| parser 難度 | 中高，內容偏自然語言 | 低，規則明確 | v2.5 現有測試優勢明顯 |
| use case | 生成 artifact、deck、prototype | 跨行業 prompt + 圖像生成/edit | 不完全同型 |
| 架構依賴 | Vite + daemon + iframe workflow | 單檔 HTML + PyQt6 desktop | 不建議整包移植 |

### 2.4 整合方案三選一

#### a) 完全替換 v2.5 schema

| 項目 | 評估 |
|---|---|
| 收益 | 與 open-design 類 ecosystem 對齊，未來可匯入更多 DESIGN.md |
| 成本 | 30-50h，含 parser、GUI、migration、測試 |
| 風險 | 破壞既有 BrandSettingsTab 與 prompt injection；使用者已填的 6 區塊需遷移 |
| 判斷 | 不推薦 |

#### b) 擴充 v2.5 加 nexu 子集

| 項目 | 評估 |
|---|---|
| 收益 | 保留現有 6 區塊，加入 v3.0 真正需要的 tokens/components/motion/voice |
| 成本 | 18-28h |
| 風險 | schema 變長，GUI 需避免變成設計系統後台 |
| 判斷 | 推薦 |

建議新增 schema 子集：

```markdown
## Spacing & Layout
- spacing_base: 4px
- container_max: 1120px
- grid: 12 columns

## Components
| component | rule |
|---|---|
| Button | 8px radius, no emoji icon |
| Card | only for repeated items, not page sections |

## Motion
- duration_fast: 120ms
- easing_standard: cubic-bezier(0.2, 0, 0, 1)

## Voice & Copy
- headline: 直接、克制、可驗證
- forbidden_terms: 空泛口號, 未驗證承諾, 誇大形容
```

#### c) 不整合

| 項目 | 評估 |
|---|---|
| 收益 | 零工程風險 |
| 成本 | 0h |
| 風險 | v3.0 仍無 component/spacing/motion 記憶，品牌一致性停在顏色與字體層 |
| 判斷 | 不推薦，除非 v3.0 只做 export |

### 2.5 推薦選項 + 理由

推薦 b：擴充 v2.5 加 nexu 子集。

理由：

| 理由 | 說明 |
|---|---|
| 相容性最高 | 舊 6 區塊不動，新 section optional |
| ROI 明確 | component/spacing/voice 會直接改善 prompt 與匯出品質 |
| 測試成本可控 | 現有 parser 可擴充，不需重寫 |
| 符合 v2.5 決策 | HANDOFF 已明確寫「整碗端走 nexu-io 不適合」 |

### 2.6 工程量估計（如果做 b）

| 工作 | 工時 |
|---|---:|
| schema v3.0 文件與 migration 規則 | 3-4h |
| parser 擴充：新增 section、table/list tolerant parsing | 5-8h |
| BrandSettingsTab UI：折疊式 advanced sections | 5-8h |
| prompt injection 摘要調整 | 2-3h |
| tests：舊 schema、混合 schema、round trip、invalid value | 3-5h |
| 合計 | 18-28h |

**決策摘要：做 b，不做 a；v3.0 借鑑 nexu schema 的可用子集，不導入 71 systems 作為核心依賴。**

## 三、B：Comment mode 評估

### 3.1 概念與 UX 流程

概念來源：

- Claude Design Help Center：comments 適合 targeted component-level changes，chat 適合 structural changes。
- Open CoDesign：click element/drop pin/note，model rewrites only that region。
- SDD v2.5 §5.1：Comment mode 需要 preview hit-test、pin storage、局部 prompt patch、差異比對。

建議 UX：

```text
Preview
  -> 使用者點元素或框選區域
  -> 建立 pin
  -> 輸入 comment
  -> 系統組成局部 patch prompt
  -> 模型產生局部改寫
  -> before/after 對照
  -> accept / reject / resolve
```

schema 級別資料模型：

```json
{
  "id": "comment-001",
  "artifact_id": "draft-20260429-001",
  "target": {
    "type": "dom-element",
    "selector": "[data-fs-id='hero-cta']",
    "bbox": { "x": 0.42, "y": 0.66, "w": 0.18, "h": 0.07 }
  },
  "body": "CTA 太像廣告，改成律師事務所可接受的專業語氣。",
  "status": "open"
}
```

### 3.2 Forma Studio 跨行業客戶的實際情境

| 客戶 | 常見修改 | Comment mode 價值 | 評估 |
|---|---|---:|---|
| 律師 | 改標題語氣、刪除過度行銷字眼、調整免責聲明位置 | 高 | 常見是局部文字與語氣修正 |
| 教師 | 改單一圖表標籤、補教材步驟、降低資訊密度 | 高 | pinpoint feedback 比重寫 prompt 直覺 |
| 行銷 | 改 CTA、價格區、社群貼文重點、品牌調性 | 高 | 迭代頻繁，但也容易變成整頁重設計 |
| 設計師 | 指定 spacing、層級、component 狀態 | 中高 | 有價值，但會期待更精準的 DOM diff |
| 會計/顧問 | 改數字表格、頁尾註記、圖表標題 | 中 | 需資料正確性，不能只靠視覺 patch |
| 餐旅/活動 | 改海報局部字、日期、優惠 | 中 | 若是 raster image，需要 mask 流程而非 DOM comment |

結論：跨行業有真實價值，但應限制在「有 preview artifact 且可定位元素」的場景。對純圖片輸出，v2.5 已有 reference + mask，Comment mode 若要自動產生 mask，成本會明顯上升。

### 3.3 技術架構選項

#### a) Web 版：iframe + element picker + 局部 prompt 改寫

| 項目 | 評估 |
|---|---|
| 適用 | HTML/prototype artifact、markdown rendered preview |
| 技術 | sandboxed iframe、`data-fs-id`、bbox、comment store |
| 工時 | 36-58h |
| 風險 | 目前 `web/forma-studio.html` 主流程是 prompt 生成，不是完整 artifact editor |
| 推薦 | 條件做 |

#### b) 桌面版：QGraphicsView + click pin + 局部 mask 自動生成

| 項目 | 評估 |
|---|---|
| 適用 | 圖片 edit / inpaint |
| 技術 | image preview、pin layer、bbox to PNG alpha mask、edits endpoint |
| 工時 | 46-72h |
| 風險 | 使用者點一下不等於正確 mask；模型仍可能修改非指定區域 |
| 推薦 | v3.0 不做，留 v4 或專案客戶需求觸發 |

#### c) 不做

| 項目 | 評估 |
|---|---|
| 收益 | 保持 v2.5 穩定，避免 artifact editor 化 |
| 成本 | 0h |
| 風險 | 使用者仍要用文字描述「哪一塊」；局部迭代體驗落後 Claude Design/Open CoDesign |
| 推薦 | 不建議完全不做；建議做小範圍 prototype |

### 3.4 工程量估計（含風險）

| 子系統 | Web 版工時 | 桌面版工時 | 風險 |
|---|---:|---:|---|
| preview target model | 6-10h | 8-12h | artifact 需穩定 id |
| element/region picker | 8-12h | 10-16h | iframe sandbox / QGraphics coordinate |
| pin store + resolve state | 5-8h | 5-8h | 需和版本綁定 |
| prompt patch builder | 6-10h | 6-10h | 模型越界改動 |
| before/after diff | 8-12h | 12-18h | 需要 render/screenshot |
| tests + feature flag | 3-6h | 5-8h | E2E 測試易脆 |
| 合計 | 36-58h | 46-72h | 中高 |

### 3.5 推薦：做 / 部分做 / 不做 + 理由

推薦：部分做。

v3.0 只做 Web/HTML preview comment prototype，條件如下：

| 條件 | 要求 |
|---|---|
| feature flag | `ENABLE_COMMENT_MODE=false` 預設關閉 |
| artifact 限制 | 只支援 HTML/preview artifact，不支援任意圖片自動 mask |
| patch 限制 | comment 只產生局部 prompt patch，accept 前不覆蓋原始內容 |
| rollback | 每次套用 comment 前建立 snapshot |

不建議 v3.0 做桌面版 QGraphicsView 自動 mask，因為 v2.5 現有 ImageEditPanel 已提供手動 mask；自動化的體驗價值不夠抵消定位與 fidelity 風險。

**決策摘要：Comment mode 有價值，但 v3.0 只做 Web artifact 的 feature-flag prototype；桌面自動 mask 留 v4。**

## 四、C：多格式匯出評估

### 4.1 5 種格式各自情境（HTML/PDF/PPTX/ZIP/Markdown）

| 格式 | 使用情境 | 真需求 | 風險 |
|---|---|---:|---|
| HTML | 互動原型、內部 review、可分享單頁 | 中 | CSS sanitizer、單檔資產 inline、瀏覽器差異 |
| PDF | 律師提案、顧問報告、教材講義、客戶交付 | 高 | 字型、分頁、列印尺寸 |
| PPTX | 教師課件、行銷提案、管理層簡報 | 高 | layout fidelity、圖片裁切、可編輯性 |
| ZIP | 工程交接、素材包、行銷檔案包 | 中低 | 路徑、授權 metadata、使用者不一定知道怎麼用 |
| Markdown | prompt handoff、NotebookLM/Claude/Docs 轉貼、版本 diff | 高 | 圖片資產引用、metadata 格式需穩 |

### 4.2 跨行業使用優先順序

| 行業 | 第一格式 | 第二格式 | 理由 |
|---|---|---|---|
| 律師 | PDF | Markdown | 客戶交付、留存、審閱紀錄 |
| 教師 | PPTX | PDF | 課堂投影與講義 |
| 行銷 | PDF | ZIP | 對客提案與素材交接 |
| 設計師 | HTML | ZIP | 原型 review 與工程交接 |
| 顧問/會計 | PDF | PPTX | 報告與簡報 |
| 產品經理 | Markdown | HTML | PRD/prompt handoff 與快速 prototype |

跨行業排序：

```text
1. PDF       # 最普遍、最容易被非技術客戶接收
2. Markdown  # 最穩、最適合 prompt/metadata handoff
3. PPTX      # 高需求，但工程風險高於 PDF/Markdown
4. HTML      # 對設計/產品有用，但不是所有客戶需要
5. ZIP       # 工程/行銷交接有用，但 v3.0 不必先做
```

### 4.3 工具鏈現況

| 平台 | HTML | Markdown | PDF | PPTX | ZIP |
|---|---|---|---|---|---|
| Web 版 | 容易，template/string | 容易，plain text | 需 browser print 或 client-side lib | 需 pptxgenjs 等 lib | 可用 JSZip，但增加依賴 |
| 桌面版 | 可用 template | 容易，stdlib | Qt PrintSupport / local Chrome / reportlab | python-pptx | stdlib `zipfile` |

v2.5 現況：

- Web 版：可複製 prompt 文字，內建 4 區塊 glow 流程與 116 條 gallery。
- 桌面版：可下載/處理生成 PNG，ImageEditPanel 支援 generate/edit/reference/mask/quality。
- SDD v2.5 已預留 exporter interface，但未實作。

工具鏈建議：

| 格式 | v3.0 建議工具 | 原因 |
|---|---|---|
| Markdown | plain text writer | 零依賴，最可測 |
| PDF | 桌面優先用 Qt PrintSupport 或 reportlab；若 HTML artifact 再評估 local Chrome | 避免先引入 Playwright/Chrome 打包體積 |
| PPTX | python-pptx 或 pptxgenjs spike | 先驗證 3-5 種 layout，不承諾高度還原 |
| ZIP | stdlib `zipfile` | 等 export model 穩定後再包 |
| HTML | 單檔 template | 只有當 artifact model 成熟時才有價值 |

### 4.4 推薦優先順序

推薦：先做 PDF + Markdown，後做 PPTX，ZIP/HTML 觀望。

| 順序 | 格式 | 原因 |
|---:|---|---|
| 1 | Markdown | 最快、最穩、可保留 attribution、可 diff |
| 2 | PDF | 客戶交付價值最高 |
| 3 | PPTX | 教師/行銷需求明確，但 layout fidelity 風險高 |
| 4 | HTML | 需先有 artifact model；目前 Web 主體仍是 prompt tool |
| 5 | ZIP | 依賴 asset/package model，過早做會變雜 |

### 4.5 工程量估計

| 工作 | 工時 |
|---|---:|
| Export Model JSON schema | 4-6h |
| Markdown exporter | 4-6h |
| PDF exporter MVP | 14-24h |
| export UI + file picker + error handling | 6-10h |
| tests：snapshot、metadata、empty asset、unicode | 4-6h |
| PDF + Markdown 合計 | 28-46h |
| PPTX spike | 20-34h |
| HTML/ZIP spike | 12-22h |

schema 級別 export model：

```json
{
  "schema_version": 1,
  "project": {
    "title": "ACME 法律顧問提案",
    "industry": "legal",
    "created_at": "2026-04-29T10:00:00+08:00"
  },
  "content": {
    "prompt": "...",
    "design_memory": "DESIGN.md summary",
    "assets": [
      { "type": "image", "path": "outputs/acme-cover.png", "license": "user-provided" }
    ]
  },
  "attribution": [
    { "source": "wuyoscar/gpt_image_2_skill", "license": "CC BY 4.0" }
  ]
}
```

**決策摘要：多格式匯出值得做，但 v3.0 先交付 PDF + Markdown；PPTX 做第二階段，HTML/ZIP 不列必做。**

## 五、v3.0 整體推薦路線

### 5.1 必做（high ROI）

| 項目 | 理由 | 預估工時 |
|---|---|---:|
| DESIGN.md 擴充子集 | 改善品牌一致性，工程量可控 | 18-28h |
| Markdown export | prompt/metadata/attribution handoff 最穩 | 4-6h |
| PDF export | 跨行業客戶交付剛需 | 14-24h |
| export model | 為 PDF/PPTX/ZIP 留共同資料層 | 4-6h |
| feature flag 基礎 | v3.0 全部 opt-in，保護 v2.5 | 3-5h |

### 5.2 觀望（看用戶反饋）

| 項目 | 觸發條件 |
|---|---|
| Comment mode Web prototype | 至少 3 個實際使用者抱怨「局部修改難描述」 |
| PPTX export | 教師/行銷/顧問使用者明確要求可編輯簡報 |
| HTML export | 已有 artifact preview model，而不只是 prompt text |
| ZIP export | 有工程交接或素材包交付需求 |

### 5.3 不做（low ROI 或風險高）

| 項目 | 不做理由 |
|---|---|
| 完整替換 nexu schema | 破壞既有 parser/GUI，收益不成比例 |
| 整合 71 品牌 systems 到核心 | 授權、品牌真實性與維護成本高；且不符合跨行業工具主軸 |
| 桌面自動 mask Comment mode | 工程風險高，v2.5 已有手動 mask |
| 一次做齊 5 種 export | 測試矩陣膨脹，PDF/PPTX fidelity 會拖慢整個 v3.0 |

**決策摘要：v3.0 主線是「品牌記憶更強 + 可交付輸出」，Comment mode 與重格式匯出不應搶主線。**

## 六、不做的事與決策依據

### 6.1 哪些功能不在 v3.0 範圍

| 不做項 | 放到 |
|---|---|
| 完整 artifact editor | v4+ |
| 多人協作 comment thread | v4+ |
| Figma import/export | 另案 |
| 71 design systems 自動同步 | 另案 |
| Desktop QGraphicsView automatic inpaint mask | v4 或客製案 |
| ZIP/HTML 必做級 export | 看 v3.0 使用數據 |
| AI sliders | 不列本次三項 backlog 主範圍 |
| 多 Provider BYOK 重構 | 另案，避免與 export/comment 同 sprint |

### 6.2 為什麼不做（不是貶低，是 ROI 計算）

ROI 依據：

| 因子 | 權重 | 判斷方式 |
|---|---:|---|
| 使用者可感知價值 | 高 | 是否直接減少交付成本 |
| 對 v2.5 破壞風險 | 高 | 是否影響 116 gallery、4 tab GUI、40 pytest |
| 工具鏈不確定性 | 中 | 是否需新增重依賴或跨平台 runtime |
| 測試複雜度 | 中 | 是否需要 E2E/screenshot/per-pixel 測試 |
| 維護成本 | 中 | 是否要長期追上游 schema 或品牌庫 |

不做不是否定功能，而是避免 v3.0 變成同時改 schema、editor、renderer、packager 的高風險版本。Forma Studio 目前最明確的商業價值是跨行業 prompt 與圖像交付，不是取代 Figma/Canva/Claude Design。

**決策摘要：不做高風險平台化功能；v3.0 的成功指標應是更穩的品牌輸入與更可用的輸出格式。**

### 7.1 PLAN-sprint-3.md 撰寫範圍

## 八、與 v2.5 的相容性

### 8.1 不能破壞既有 116 條 gallery / 4 tab GUI / 24 PASS pytest

需保護的基線：

| 基線 | 來源 | v3.0 要求 |
|---|---|---|
| 116 條 gallery / 17 類 / 雙 source | README、DEVLOG、web inline gallery | 不改 gallery schema；新增 export 只讀取 |
| Web 4 區塊 glow 流程 | `web/forma-studio.html` | 不改原流程完成條件 |
| 桌面 4 tab GUI | DEVLOG headless smoke | 新功能不得改 tab 啟動 |
| ImageEditPanel reference/mask/quality | `desktop/app/widgets/image_edit_panel.py` | 不改 edits endpoint 行為 |
| DESIGN.md 6 區塊 | `desktop/app/utils/design_memory.py` | v3 section 必須 optional |
| pytest | README 寫 24 PASS，DEVLOG 最新為 40 PASS | v3.0 應以最新 40 PASS 為實際門檻，至少不能低於 README 宣告的 24 PASS |

### 8.2 feature flag 策略（v3.0 全部 opt-in）

建議 flags：

```text
ENABLE_DESIGN_MD_V3=false
ENABLE_EXPORT_MARKDOWN=false
ENABLE_EXPORT_PDF=false
ENABLE_EXPORT_PPTX=false
ENABLE_COMMENT_MODE=false
```

相容策略：

| 功能 | 預設 | fallback |
|---|---|---|
| DESIGN.md v3 section | 關 | 只讀 v2.5 6 區塊 |
| Markdown export | 關 | 保留複製 prompt |
| PDF export | 關 | 保留 PNG/文字輸出 |
| PPTX export | 關 | 不顯示入口 |
| Comment mode | 關 | 保留手動 prompt 與 mask workflow |

最小驗收清單：

```text
v3.0 compatibility gate
├── Web
│   ├── gallery total remains 116
│   ├── 4 section flow still completes
│   └── feature flags off equals v2.5 behavior
├── Desktop
│   ├── 4 tabs launch
│   ├── ImageEditPanel still blocks edit without reference
│   ├── DESIGN.md v2.5 round trip still passes
│   └── export flags off hides new UI
└── Tests
    ├── existing desktop tests pass
    ├── new parser tests pass
    └── export snapshot tests pass
```

**決策摘要：v3.0 全部 opt-in；任何功能只要破壞 gallery、4 tab、ImageEditPanel 或 DESIGN.md v2 parser，就不應合入。**

|---:|---|---|
| P0 | DESIGN.md v2.5 schema 擴充為 nexu 子集 | 做 |
