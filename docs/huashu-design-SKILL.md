## 核心原則 #0：事實驗證先於假設（最高優先級）

任何涉及具體產品/技術/事件/人物的存在性、發布狀態、版本號、規格參數，
**第一步必須 WebSearch 驗證，禁止憑訓練語料做斷言。**

禁止句式：
- ❌ "我記得 X 還沒發布"
- ❌ "X 目前是 vN 版本"（未搜索的斷言）
- ✅ "我先 WebSearch 確認 X 的最新狀態"

## 核心資產協議（完整 5 步）

### 資產優先級（識別度排序）
| 資產類型 | 識別度 | 必需性 |
|---------|-------|-------|
| Logo | 最高 | 任何品牌都必須有 |
| 產品圖/渲染圖 | 極高 | 實體產品必須有 |
| UI 截圖/介面 | 極高 | 數位產品必須有 |
| 色值 | 中 | 輔助 |
| 字型 | 低 | 輔助 |

### Step 1 · 問（資產清單一次問全）
```
關於 <brand/product>，你手上有以下哪些資料？按優先級列：
1. Logo（SVG / 高清 PNG）—— 任何品牌必備
2. 產品圖 / 官方渲染圖 —— 實體產品必備
3. UI 截圖 / 介面素材 —— 數位產品必備
4. 色值清單（HEX / RGB）
5. 字型清單（Display / Body）
6. Brand guidelines / Figma / 品牌官網連結

有的直接發我，沒有的我去搜/抓。
```

### Step 2 · 搜官方渠道
| 資產 | 搜索路徑 |
|-----|---------|
| Logo | `<brand>.com/brand` · `/press` · `/press-kit` · 官網 header inline SVG |
| 產品圖 | 官網產品頁 hero · 官方 YouTube launch film 截幀 · 官方新聞稿 |
| UI 截圖 | App Store/Google Play 截圖 · 官網 screenshots · 演示視頻截幀 |
| 色值 | 官網 inline CSS · Tailwind config · brand guidelines PDF |

### Step 3 · 下載資產（三條兜底）
**Logo（任何品牌必需）：**
1. 獨立 SVG/PNG 文件（最理想）
2. 官網 HTML 全文提取 inline SVG（curl 後 grep `<svg>`）
3. 官方社交媒體 avatar（最後手段）

**產品圖（實體產品必需）：**
1. 官方產品頁 hero image
2. 官方 press kit
3. YouTube launch video 截幀（yt-dlp + ffmpeg）
4. Wikimedia Commons
5. AI 生成兜底（以官方參考圖為基底）

**UI 截圖（數位產品必需）：**
1. App Store / Google Play 產品截圖
2. 官網 screenshots section
3. 演示視頻截幀

### Step 4 · 驗證 + 提取色值
```bash
grep -hoE '#[0-9A-Fa-f]{6}' assets/<brand>-brand/*.{svg,html,css} \
  | sort | uniq -c | sort -rn | head -20
```
過濾黑白灰，識別品牌高頻色。

**警告**：截圖裡的演示品牌色 ≠ 目標品牌色，務必區分。

### Step 5 · 固化 brand-spec.md
```markdown
# <Brand> · Brand Spec
> 採集日期：YYYY-MM-DD

## 核心資產（一等公民）
### Logo
- 主版本：`assets/<brand>/logo.svg`
- 反色版：`assets/<brand>/logo-white.svg`

### 產品圖（實體產品）
- 主視角：`assets/<brand>/product-hero.png`
- 細節圖：`assets/<brand>/product-detail-*.png`

### UI 截圖（數位產品）
- 主頁：`assets/<brand>/ui-home.png`
- 核心功能：`assets/<brand>/ui-feature-<n>.png`

## 輔助資產
### 色板
- Primary: #XXXXXX
- Background: #XXXXXX
- Accent: #XXXXXX

### 字型
- Display: <font>
- Body: <font>

### 氣質關鍵詞
- <3-5 個形容詞>
```

所有 HTML 引用 `var(--brand-*)` CSS 變量，不允許 CSS 剪影代替真實資產。

### 5-10-2-8 品質門檻（鐵律）
- **5 輪搜索**：多渠道交叉搜
- **10 個候選**：至少 10 個才開始篩選
- **選 2 個好的**：精選最終素材
- **每個 8/10 以上**：不夠分寧可不用，用誠實 placeholder

評分維度：分辨率 ≥2000px / 版權清晰 / 品牌氣質契合 / 構圖一致性 / 獨立敘事能力

**Logo 例外**：有就必須用，不走 5-10-2-8。Logo 是識別度根基，就算 6 分也比沒有強 10 倍。

## 設計方向顧問（Fallback 模式，完整 8 Phase）

### 觸發條件
- 需求模糊（"做個好看的"、沒有具體參考）
- 主動要求"推薦風格"、"給幾個方向"
- 沒有任何 design context

### 8 個 Phase

**Phase 1 · 深度理解需求**
提問（一次最多 3 個）：目標受眾 / 核心信息 / 情感基調 / 輸出格式

**Phase 2 · 顧問式重述（100-200字）**
重述本質需求、受眾、場景、情感基調。以「基於這個理解，我為你準備了 3 個設計方向」結尾。

**Phase 3 · 推薦 3 套設計哲學（必須差異化）**
- 必須來自 3 個不同流派
- 含設計師/機構名（如「Kenya Hara 式東方極簡」）
- 50-100 字解釋「為什麼這個設計師適合你」
- 3-4 條標誌性視覺特徵 + 3-5 個氣質關鍵詞

**差異化規則**：3 方向必須來自不同流派形成明顯視覺反差
| 流派 | 視覺氣質 |
|-----|---------|
| 信息建築派（01-04） | 理性、數據驅動、克制 |
| 運動詩學派（05-08） | 動感、沉浸、技術美學 |
| 極簡主義派（09-12） | 秩序、留白、精緻 |
| 實驗先鋒派（13-16） | 先鋒、生成藝術、視覺衝擊 |
| 東方哲學派（17-20） | 溫潤、詩意、思辨 |

**Phase 4 · 展示預製 Showcase 畫廊**
匹配現有場景樣例讓用戶先看

**Phase 5 · 生成 3 個視覺 Demo（並行或串行）**
- 使用用戶真實內容（不是 Lorem ipsum）
- 截圖展示 3 個方向的視覺差異

**Phase 6 · 用戶選擇**
選一個深化 / 混合（"A 的配色 + C 的布局"）/ 微調 / 重來

**Phase 7 · 生成 AI 提示詞**
結構：`[設計哲學約束] + [內容描述] + [技術參數]`
- 用具體特徵（寫 hex 色碼，不寫「極簡」這種虛詞）
- 包含顏色、比例、空間分配、輸出規格

**Phase 8 · 進入主幹 Junior Designer 流程**

## 20 種設計哲學完整庫

### 流派一：信息建築派（01-04）
視覺氣質：理性、數據驅動、克制、有序

01 **Pentagram 信息建築**
- DNA：網格系統 · 資訊層次嚴謹 · 文字排版精確
- 代表：Michael Bierut 的 NYC Identity
- 視覺特徵：清晰的信息層級 · 精確的網格對齊 · 克制的色彩使用
- 氣質關鍵詞：理性 · 可信 · 精準 · 專業

02 **Field.io 系統數據**
- DNA：數據可視化 · 算法美學 · 動態生成
- 視覺特徵：數據驅動圖形 · 清晰的信息架構 · 精確的網格系統
- 氣質關鍵詞：科學 · 精確 · 動態 · 前沿

03 **Bloomberg 新聞排版**
- DNA：新聞版面設計 · 信息密度高 · 快速掃描
- 視覺特徵：嚴格的欄位系統 · 強調數據 · 清晰的視覺層次
- 氣質關鍵詞：權威 · 即時 · 高效 · 可信

04 **IDEO 功能主義**
- DNA：以人為本 · 功能優先 · 清晰的用戶路徑
- 視覺特徵：直覺化的視覺語言 · 清晰的操作引導 · 包容的設計
- 氣質關鍵詞：友好 · 實用 · 包容 · 人性化

### 流派二：運動詩學派（05-08）

07 **生成藝術派**

- 視覺特徵：規律性的複雜圖案 · 數學驅動的美學 · 獨特的生成紋理
- 氣質關鍵詞：獨特 · 算法 · 無限 · 生成

08 **技術美學派**

- 視覺特徵：機械精度感 · 工業材質感 · 技術細節展示

## Junior Designer 工作流

1. 開工前一次性列出所有假設 + 待確認事項，等用戶批量回答
2. HTML 裡先寫 assumptions + placeholders + reasoning comments
3. 盡早 show 給用戶（哪怕只是灰色方塊）
4. 填充真實內容
5. 推出 Tweaks 面板（可調：配色/字級/密度）
6. Playwright 測試交互

## 反 AI Slop 完整規則

### 為什麼要反 slop
AI slop = AI 訓練語料裡最常見的「視覺最大公約數」
- 用戶請你設計，是要**他的品牌被認出來**
- AI 默認產出 = 所有品牌混合 = 沒有任何品牌被認出來
- 反 slop 不是審美潔癖，是**替用戶保護品牌識別度**

### 禁止元素（含原因）
| 元素 | 為什麼是 slop | 例外 |
|-----|------------|-----|
| 激進紫色漸變 | AI「科技感」萬能公式，SaaS/AI/web3 每個落地頁都有 | 品牌本身用 |
| Emoji 作圖標 | 每個 bullet 都配 emoji，「不夠專業就用 emoji 湊」 | 品牌本身用（如 Notion） |
| 圓角卡片+左彩色 border accent | 2020-2024 Material 爛大街組合，已成視覺噪音 | 用戶明確要求 |
| SVG 畫人臉/場景 | AI 畫的 SVG 人物永遠五官錯位 | 幾乎沒有 |
| CSS 剪影代替產品圖 | 任何實體產品都長一樣，品牌識別度歸零 | 幾乎沒有 |
| Inter/Roboto/Arial 做 display | 太常見，看不出「有設計的產品」 | 品牌 spec 明確用 |
| 深藍底 #0D1117 | GitHub dark mode 爛大街復刻 | 開發者工具品牌 |

### 正向做法
- ✅ `text-wrap: pretty` + CSS Grid + oklch()
- ✅ 用 spec 裡已有的色，不臨場發明新顏色
- ✅ 中文用「」引號不用 ""
- ✅ 一個細節做到 120%，其他做到 80%

## 5 維度專家評審

| 維度 | 評分標準（0-10）|
|-----|--------------|
| 哲學一致性 | 設計語言是否一以貫之？各元素服從同一哲學體系？ |
| 視覺層級 | 眼睛知道先看哪裡？資訊重要性對應視覺重量？ |
| 細節執行 | 間距/字間距/對齊/色彩是否到位？ |
| 功能性 | 是否真正解決溝通問題？受眾能快速理解？ |
| 創新性 | 有讓人眼睛一亮的決策嗎？還是「做完了」？ |

輸出格式：
- 雷達圖可視化
- KEEP（做得好，保留）
- FIX（需要修復，格式：將A改為B，因為C）
- QUICK WINS（3分鐘可改的快速提升）
- 反 AI Slop 診斷
- 達到 9 分最關鍵的一個改變
